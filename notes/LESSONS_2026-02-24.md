# LESSONS_2026-02-24.md — Implementation and Testing Observations

This note records what we learned while implementing and hardening DNA VisiCalc (core + file + TUI), and puts that in context of the bigger `..\Foundation` program.

## 1. What Worked Well

1. Clean layering paid off immediately.
- Splitting into `dnavisicalc-core`, `dnavisicalc-file`, and `dnavisicalc-tui` made each layer independently testable and refactorable.
- This directly matches Foundation `CONSTR-002` (I/O outside core) and reduced regression risk.

2. Deterministic behavior is a force multiplier.
- Stable eval ordering, explicit cycle errors, and explicit recalc modes made bugs reproducible and testable.
- This validates the Foundation "determinism-first debugging" doctrine in practice.

3. TUI reducer/state-machine style is the right shape.
- Treating UI as state transitions (`Action -> App`) made scripted tests and fuzzing practical.
- The test harness could drive behavior without a real terminal for most logic.

4. File format discipline matters even for a toy.
- A strict, line-oriented format with clear parse errors gave good failure diagnostics.
- Broken-file recovery tests prevented "bad input kills session" behavior.

5. Property/fuzz testing found quality gains faster than example-only tests.
- Parser, command, and action fuzzing revealed edge assumptions that normal unit tests missed.
- Round-based hardening delivered measurable reliability improvements.

## 2. Main Concerns and Limits Observed

1. 100% coverage is not free.
- We reached high coverage (~89% lines/functions), but terminal runtime plumbing remains hard to cover without pseudo-terminal integration.
- Chasing 100% with low-value tests can waste effort unless seams are designed first.

2. Edge semantics are still under-specified.
- Duplicate cell records in files, numeric text canonicalization, parser nesting limits, and command quoting are all policy choices not yet fully specified.
- Tests exposed these as specification gaps, not only implementation gaps.

3. UI safety and UX policy need explicit decisions.
- Current behavior is robust, but details like quit-in-edit semantics, status history, and edit replacement vs append need normative rules.

4. Scope creep pressure appears quickly.
- Even a minimal TUI + file layer introduces many side paths (runtime loops, path handling, malformed inputs).
- This supports the Independent Review warning: implementation must stay narrow and evidence-driven.

## 3. Foundation Context: What This Pathfinder Suggests

1. The Independent Review diagnosis was correct: code-first is healthier.
- Rapid implementation + test/fix loops produced concrete artifacts and clearer requirements than further abstract planning would have.

2. The Foundation architecture direction is viable, but sequencing is critical.
- Hard boundaries and deterministic state were easy to apply now.
- Dual-engine, broad formal stack, and full protocol surface should remain staged, not front-loaded.

3. "Regressions are assets" is practical, not rhetorical.
- Every discovered bug was easiest to stabilize by adding targeted tests and keeping them.
- This should scale into Foundation-style trace/minimized corpus discipline.

4. Specification pressure should come from failing tests and edge cases.
- The best next spec clarifications are those surfaced by malformed files, fuzzing, and UI action sequences.

## 4. Recommended Near-Term Moves

1. Add an explicit minimal OpLog in pathfinder v-next.
- Keep the current engine, but route mutations through an operation record to start aligning with Foundation boundary #1.

2. Add deterministic replay artifacts.
- Persist scripted UI/file/action traces as reusable regression artifacts.

3. Introduce a pseudo-terminal integration harness.
- Close runtime coverage gaps and validate real event loop behavior end-to-end.

4. Decide and document edge policies now.
- Duplicate file records, numeric serialization format, parser depth guard, quoted command paths, and edit semantics.

5. Keep formal-method hooks small and executable.
- Prefer invariant checks and replayable traces now; stage heavier proof/oracle work after semantics are stabilized by implementation evidence.

## 5. Open Questions Worth Carrying Forward

1. Should file loading be strict-reject on duplicates, or deterministic last-write-wins with diagnostics?
2. Should number serialization be canonicalized for stable cross-platform textual diffs?
3. Should formula parsing enforce a maximum nesting depth to prevent pathological inputs?
4. Should command parsing support quoted arguments for file paths with spaces?
5. Should quitting while editing require explicit confirmation to avoid accidental loss?
